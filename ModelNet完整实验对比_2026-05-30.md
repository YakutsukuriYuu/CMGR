解读：

- `Base route rate` 应该越接近 1 越好；Task 2 只有 `0.560`，说明大量 base 样本没有走 frozen NetB。
- `Novel route rate` 应该越接近 0 越好；Task 3/4 约 `0.21~0.22`，说明约五分之一 novel 样本被错误路由到 base 分支。
- BND 仍然是增量阶段明显风险点，但它不是唯一问题；base 只有 `75.12%` 已经决定了上限很低。

## 结论

本次完整实验仍明显低于论文 M2M。

最关键的问题是：

1. Base 阶段只有 `75.12%`，比论文 `95.00%` 低 `19.88` 个百分点。
2. 增量任务从 Task 1 开始就明显低：`53.60%` vs 论文 `84.80%`。
3. 当前 `num_views=10`，论文记录为 `20`；这是已知不等效项。
4. 当前 BND 路由不够稳定，尤其 Task 2 base route rate 只有 `0.560`。
5. Base 训练最后 train acc 约 `79%`，不是明显过拟合，而是整体表征/实现存在缺口。

下一步优先排查方向：

1. 对齐论文的 20-view 渲染，或确认论文是否实际使用了 FILP-3D/CLIP2Point 的 10-view 配置。
2. 检查当前 DepthEncoder 训练范围是否与论文一致；当前代码 base 阶段训练整个 CLIP visual，增量阶段冻结 DepthEncoder。
3. 检查分类 logits 是否与论文一致；当前训练时只用 `geo_sim`，eval 才加 `CLIP(I_E,d)`。
4. 检查 ReCon/CLIP2Point 权重加载是否完整，尤其 CLIP2Point checkpoint 与当前 wrapper 的 key 是否真正匹配。
5. 单独评估 base 阶段 `geo_sim` 与 `geo_sim + clip_sim` 的差异，确认 eval 加 CLIP 图像分支是否帮到忙。
